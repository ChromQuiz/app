# CIQ Customer Experience Journeys

CIQ の UI/UX/CX を「今どこにいるか・何が安全か・次に何をすべきか」で設計するための根拠。
4 ペルソナの主要ジャーニーと、各画面が解決すべき課題を定義する。エージェントも人間もこの文書を設計判断の出発点にする。

## 設計主軸（再掲）

- **デザインシステムの正典は `design-system/MASTER.md`**（2026-07 全面再構築 / Apple方向の白・黒・グレー基調）。トークン・コンポーネント・タイポグラフィ・メール設計はそちらを参照。
- **色**: Background `#FFFFFF` / Text `#1A1A1E` / Sub `#6E6E76` / Border `#E6E6EA` / Accent `#0B57D0`(CTA・リンク限定) / Success `#1E7A44` / Warning `#9A6200` / Error `#C22945` / Gold(成績のみ) `#9A6A00`。ライト/ダーク両対応。紫・グラデ・グロー・ガラス風は禁止。
- **タイポ**: システムフォントスタック(SF Pro前提)。mono(SF Mono系)は数値・受付番号・ID・コード専用。
- **状態は色だけで伝えない**: 色 + アイコン + テキスト + 配置の複合。44px touch / focus ring 3px / prefers-reduced-motion 対応。
- **ページ構成**: 参加者のセルフ操作は my.html(マイエントリー)に集約。edit/cancel/late/disclosure は廃止。admin はフェーズタイムライン。詳細は `docs/page_structure.md`。

## 1. 主催者（大会運営者）

### ジャーニー
1. **作成**: `index.html#create` でプロジェクト作成 → 採点者招募情報（ID・パスワード）を受け取る
2. **エントリー公開**: `admin.html` Entry フェーズで参加者向けページを公開。公開期間・定員・キャンセル繰上期限を設定
3. **参加者管理**: 参加者一覧の確認、手動追加、名簿 CSV 出力
4. **PDF 生成**: Materials フェーズで解答用紙 PDF を生成・配布
5. **答案取込**: Answers フェーズで答案 PDF を一括スキャン・保存
6. **採点監視**: Judging フェーズで採点進捗・要確認件数を監視。衝突があれば解決を促す
7. **衝突解決**: `conflict.html` で採点者間の意見不一致を確定
8. **成績照会公開**: Results フェーズで成績照会ページを公開。CSV/PDF 出力

### 各画面が解決すべきこと
- **index**: 「何をすべきか」が3秒で分かる。作成導線は `#create` に隠し、参加者導線に出さない
- **admin**: 今の大会状態（エントリー中/採点中/公開済等）、次にやること、危険な未処理（要確認数・未公開 等）、公開中の参加者向けページが一画面で把握できること
- **conflict**: 未解決/解決済みの判別、票の内訳、模範解答との対比、確定操作が迷わないこと

### 主催者の「不安」を減らすポイント
- 公開/停止の切り替え状態が常に見える（公開中かどうかの不安）
- 危険な未処理（要確認多数 等）が目立つ（見落とし不安）
- 採点進捗が定量的（いつ終わるかの不安）
- 成績公開前の未確定項目チェック（誤公開不安）

## 2. 採点者

### ジャーニー
1. **参加**: `index.html` でプロジェクト ID + パスワードで入室
2. **問題選択**: `judge.html` で担当する問題を選ぶ。満員・自分担当・未着手を即判別したい
3. **採点**: `question.html` で答案画像を見て正解/不正解/保留を判定
4. **完了確認**: 自分の担当問題の完了状況を確認
5. **次の問題**: 別の問題へ移動

### 各画面が解決すべきこと
- **index**: 採点者参加が迷いなくできること。参加後の次のアクション（問題一覧へ）が明示されること
- **judge**: 未着手/自分担当/満員/完了/要確認が即判別できるステータスボードであること。担当可能な問題が目立つこと
- **question**: 答案画像が大きく見えること。模範解答と対比できること。正解/不正解/保留が押しやすいこと。進捗・残りが分かること。キーボード操作が分かること

### 採点者の「不安」を減らすポイント
- どの問題が自分担当か明確（二重採点・取り忘れ不安）
- 満員で入れない問題が即分かる（無駄なクリック不安）
- 今何問目/残り何問か分かる（終了見通し不安）
- 採点結果の送信状態（保存中/保存済）が分かる（送信漏れ不安）

## 3. 参加者（コンテスタント）

### ジャーニー
1. **エントリー**: `entry.html` でメール認証 → 登録フォーム → 受付番号・QR 発行
2. **メール確認**: 控えメール（受付番号・QR・パスワード）を受信・保存
3. **セルフ操作**: `my.html`(マイエントリー)で1回認証 → 確認 / QR再表示・保存 / 編集 / 遅刻連絡 / キャンセル
4. **当日受付**: `checkin.html`（スタッフがQRを読み取る形式だが、参加者はQRを提示する側）
5. **成績照会**: `my.html` の成績セクション(照会公開期間のみ表示)で自分の成績・順位を確認

### 各画面が解決すべきこと
- **entry**: 何のための入力か、入力後どうなるか、完了後に何を保存すべきか（メール/QR/受付番号）が明確であること。ステップ感と送信状態が分かること
- **my**: 1回の認証で全セルフ操作が完結すること。受付番号+QRが主役。危険操作（キャンセル）は最下部に分離し影響範囲を明記
- **my(成績)**: スコア・順位・内訳が分かりやすいこと。共有端末向けにログアウト導線があること

### 参加者の「不安」を減らすポイント
- 送信中に操作できなくする（二重送信不安）
- 成功後に「メールを保存してください」が強く目立つ（当日不安）
- 受付番号が大きく表示される（見失不安）
- キャンセルが取り消せないことが操作前に分かる（誤操作不安）

## 4. 当日受付スタッフ

### ジャーニー
1. **カメラ起動**: `checkin.html` を開き、カメラを許可
2. **読取**: 参加者のQRをカメラにかざす
3. **判定**: 受付完了/受付済み/キャンセル待ち/キャンセル済/エラー を即座に判別
4. **例外対応**: エラーやキャンセル済みの場合、参加者を案内

### 各画面が解決すべきこと
- **checkin**: 受付机のラップトップに参加者が向かってQRをかざす前提。カメラ領域(かざす枠つき)と判定結果の両方が大きいこと。色 + アイコン + テキスト + 番号で判別できること。済/未/計はヘッダーで控えめに常時表示。履歴リストは置かない(名簿確認は admin 側)

### 受付スタッフの「不安」を減らすポイント
- 成功/失敗が一瞬で分かる（列を滞留させない）
- 「受付済み」を見逃さない（二重受付防止）
- エラー時に次に何を言えばよいか分かる（案内文言）
- 受付状況がリアルタイム（進捗不安）

## 共通設計原則（全ペルソナ）

1. **Next Best Action**: 全画面で「次に押すべきボタン」が常に見える
2. **状態は複合で伝える**: 色 + アイコン + テキスト + 配置。色のみ禁止
3. **async は4状態**: loading / success / error / recovery。UI を凍結させない
4. **空状態は行動を示す**: 「データがありません」だけでなく「次に〇〇してください」
5. **エラーは回復可能に**: 「何が起きたか」「次に何をするか」を書く
6. **危険操作は影響範囲を明記**: confirmation だけでなく「〇〇件に影響します」等
7. **戻る操作を壊さない**: URL hash/query による deep link、ブラウザ back の保存
8. **キーボード・スクリーンリーダー対応**: tablist/aria、focus-visible 3-4px、semantic HTML
9. **モバイル 375px 最低**: 表は横スクロール or カード化、touch target 44px+

## 用語（厳守）

- 「エントリー」: 登録・エントリーフォーム関連
- 「当日受付」: 大会当日の受付フロー
- 「受付番号」: 参加者番号のみに使用
- 「成績照会」: 成績開示ページ（「成績開示」と混用しない）
